# v0.20 — `layers_redraw` mask-aware routing

## Context

`mcp__vulca__layers_redraw` produces wrong-subject output for sparse-alpha layers. Live evidence from the IMG_6847 iPad-cartoon dogfood (`docs/visual-specs/2026-04-27-ipad-cartoon-roadside/`) on 2026-04-27:

| layer | area_pct | original alpha-masked | redrawn alpha-masked | drift |
|---|---|---|---|---|
| flower_cluster_c | 3.3% | RGB(105,113,79) hue=74° L=0.38 (white-on-green) | RGB(98,149,75) hue=101° L=0.44 (green leaves) | white→green |
| flower_cluster_c retry | 3.3% | (same) | RGB(137,121,94) hue=37° L=0.45 (brown ground) | white→brown |
| yellow_truck v1 | 0.68% | RGB(178,177,148) hue=57° (yellow) | RGB(109,166,139) hue=151° L=0.54 (green) | yellow→green |
| red_car | 2.58% | dark red | RGB(155,119,122) hue=354° + 95 unique color buckets | shape-fail: model painted multi-color flag/banner |

For DENSE layers (sky 24%, trees 6%, hedge 49%) — redraws worked. The same bug was hit on Scottish 2026-04-25 (`docs/visual-specs/2026-04-23-scottish-chinese-fusion/decompose/v0_17_14_native/NOTES.md`) on the row-of-6-lanterns case (8.05% area, multi-instance) and was sidestepped via stand-alone `generate_image`. IMG_6847 has no sidestep available because additive layered composition needs precise per-object spatial control that single-frame `generate_image` cannot provide.

**This spec is the architectural fix. v0.18 backlog item `feedback_cream_flat_reference_limit` is closed by this spec.**

## A. Diagnosis verification

The bug: `redraw_layer` (`src/vulca/layers/redraw.py:211-385`) flattens the alpha-sparse layer onto a cream RGB canvas (`_flatten_layer:106-131`), sends THAT as `reference_image_b64` to `OpenAIImageProvider.generate` (`src/vulca/providers/openai_provider.py:151-365`), which posts to `/v1/images/edits` with `files={"image": ref}` only — **no mask file**. This is unmasked img2img: the model freely paints the whole canvas, then `preserve_alpha=True` post-hoc crops back to the original sparse bbox. For sparse alpha, the model has no spatial cue for "where the subject is" → crop yields whatever was painted at those coords.

**The fix is already in the codebase but not wired**: `OpenAIImageProvider.inpaint_with_mask` (`openai_provider.py:367-500`) calls the same endpoint WITH a mask file. Production-ready, used by `inpaint_artwork` API at `src/vulca/inpaint.py:217-315`. We wire it into `layers_redraw` for sparse-alpha layers.

### Architectural choice locked: option (b)

The mask-aware path uses **`image=flattened-layer-on-cream` + `mask=inverse-of-layer-alpha`**, NOT `image=manifest-source-photo`.

Reasoning: `redraw_layer` is semantically a *layer* operation. Using `manifest.source_image` as the inpaint base would duplicate `inpaint_artwork` semantically (`mcp_server.py:570-629`) — it'd raise the question "why isn't this just `inpaint_artwork`?" Option (b) keeps the layer semantics and removes the "source file deleted/moved" failure mode entirely.

## B. Surgical fix spec

### B1 — Sparsity predicate (live alpha, not manifest)

```python
# In redraw_layer, AFTER reading src_rgba and BEFORE _flatten_layer
import numpy as np
alpha = src_rgba.split()[-1]                    # L-mode band
arr = np.array(alpha)
visible_px = int((arr > 0).sum())
total_px = arr.size
area_pct = 100.0 * visible_px / total_px
rows = (arr > 0).any(axis=1); cols = (arr > 0).any(axis=0)
bbox_h = int(rows.sum()); bbox_w = int(cols.sum())
bbox_fill = visible_px / (bbox_h * bbox_w) if (bbox_h * bbox_w) else 0.0

# Predicate catches BOTH:
#   - IMG_6847 single-region sparse (area_pct=3.3%, bbox_fill≈0.6 — caught by area_pct)
#   - Scottish multi-instance sparse (area_pct=8.05%, bbox_fill<0.2 — caught by bbox_fill)
sparse = (area_pct < 5.0) or (bbox_fill < 0.5)
```

Numpy already in deps (`pyproject.toml:33`). Existing module `src/vulca/layers/blend.py:160` imports numpy — adding to `redraw.py` is bit-for-bit fine.

### B2 — RGBA mask construction (polarity-correct, no blur)

```python
from PIL import ImageOps

subject = alpha.point(lambda v: 255 if v > 0 else 0)   # 255 where subject is
edit_mask_channel = ImageOps.invert(subject)            # 0 where subject (EDIT), 255 where empty (PRESERVE)
mask = Image.new("RGBA", alpha.size, (0, 0, 0, 0))
mask.putalpha(edit_mask_channel)
# Save to temp file as RGBA PNG; NO GaussianBlur — {0,255} hard binary only
```

**Invariant**: after construction, `mask.split()[-1]` has value 0 over subject pixels and 255 over empty pixels — this is the OpenAI edit/preserve convention applied correctly. The unit test in D1.2 pins this invariant.

### B3 — Inpaint base image construction

The `image` argument to `inpaint_with_mask()` is the existing **flattened layer** (`flat_img` from `_flatten_layer` at `redraw.py:296-320`, cream + subject). Already constructed in `redraw_layer`'s current path; reuse directly. **No `manifest.source_image` lookup. No new resolver.**

### B4 — API surface (auto-route at predicate gate)

```python
def redraw_layer(
    target, prompt,
    *,
    route: Literal["auto", "img2img", "inpaint"] = "auto",
    ...
):
    sparse = _compute_sparsity(target.alpha)  # B1

    if route == "auto":
        chosen_route = "inpaint" if (sparse and hasattr(provider, "inpaint_with_mask")) else "img2img"
    else:
        chosen_route = route
        if chosen_route == "inpaint" and not hasattr(provider, "inpaint_with_mask"):
            logger.warning(
                "Provider %s lacks inpaint_with_mask; falling back to img2img.",
                type(provider).__name__,
            )
            chosen_route = "img2img"

    if chosen_route == "inpaint":
        return await _redraw_via_inpaint_mask(target, prompt, provider, ...)
    return await _redraw_via_img2img(target, prompt, provider, ...)  # existing flow extracted
```

**v0.20 default = `auto`** — predicate gate IS the safety. Dense layers (the vast majority of existing callers) hit `sparse=False` and stay on img2img bit-identical. Sparse layers route to inpaint when capable, fall back to img2img on incapable providers. No silent "ship the bug as default" hedging.

This reverses codex v2's conservative `route="img2img"` default after reviewer round 2 argued: opt-in-only ships a known bug as default for the case the fix is FOR. The predicate already gates the change; the capability fallback handles non-OpenAI providers; the burn risk from `feedback_layers_redraw_inplace_overwrite` (silent destructive defaults) does not apply here because the new path is non-destructive (different output filename via `output_layer_name`).

MCP signature: `layers_redraw(layer_id, prompt, route="auto", ...)` — additive kwarg, schema-compatible.

### B5 — Prompt semantics for inpaint route

Replace `_build_redraw_prompt`'s canvas-guard ONLY for `chosen_route == "inpaint"`:

```python
INPAINT_PROMPT_PREFIX = (
    "Repaint the transparent region of the mask (where mask alpha=0) as: {instruction}. "
    "Leave the opaque region (mask alpha=255) bit-identical. "
    "Keep the same silhouette, placement, and scale as the underlying subject. "
    "Do not bleed color outside the mask boundary."
)
```

This wording is reviewer-corrected: "the masked subject pixels" is ambiguous (gpt-image-2 docs treat alpha=0 as "the part to inpaint" not "the masked part"). Explicit alpha-channel reference removes ambiguity.

img2img path prompt unchanged.

### B6 — Capability gate

```python
if route == "inpaint" or (route == "auto" and sparse):
    if not hasattr(provider, "inpaint_with_mask"):
        # explicit-inpaint: warn, fall back; auto: silent fall back
        chosen_route = "img2img"
        if route == "inpaint":
            logger.warning("Provider %s lacks inpaint_with_mask; falling back to img2img.", type(provider).__name__)
```

`hasattr` covers mock + gemini + comfyui + any future provider. No name-string matching. Existing model-family gate inside `inpaint_with_mask` (`openai_provider.py:387-391`) handles `gpt-image-*` enforcement at the provider layer.

### B7 — `redraw_merged` DEFERRED to v0.21

`redraw_merged` (`redraw.py:416-519`) calls `provider.generate(reference_image_b64=...)` with no mask — same shape, parallel pattern. **But the fix is NOT parallel** (reviewer round 2 finding):

`blend_layers` (`blend.py:168-219`) composites N selected layers over a transparent canvas, producing a UNION alpha. For typical merged sets (e.g. `flower_cluster_a + flower_cluster_b + flower_cluster_c`), the union alpha is dense → `sparse=False` → predicate doesn't fire → no fix. For overlapping layer sets (subject + background), dense again. Only genuinely-sparse merged sets (two 3% layers) would trigger inpaint, but the prompt "redraw the masked subject" makes no semantic sense for a disjoint union of unrelated subjects.

**Decision**: ship v0.20 with `redraw_merged` ON STATUS QUO. Do not add `route` kwarg to `redraw_merged` in v0.20. Document in CHANGELOG: "merged-redraw retains v0.18 behavior; v0.20 mask-aware fix applies to single-layer redraw only. Merged-redraw redesign tracked for v0.21."

### B8 — Image+mask dimension handling (NEW per reviewer round 2 #3)

`flower_cluster_c.png` is 4032×3024. OpenAI `/v1/images/edits` `size` parameter accepts only `{1024×1024, 1024×1536, 1536×1024}`. The provider's `inpaint_with_mask` defaults to `1024×1024` when input size is unsupported (`openai_provider.py:407-415`).

**Implication**: feeding the layer at native size would upload 4032×3024 bytes but receive a 1024×1024 result. The mask must match the upload bytes — also 4032×3024 — but then the result is at API-default size. preserve_alpha re-application against the original 4032×3024 alpha would either fail size-mismatch or trigger an upsample with quality loss.

**Resolution**: in `_redraw_via_inpaint_mask`:

```python
# 1. Pick API-supported size matching layer aspect ratio
def _pick_inpaint_size(w: int, h: int) -> tuple[int, int]:
    aspect = w / h
    if aspect > 1.3:    return (1536, 1024)
    if aspect < 0.77:   return (1024, 1536)
    return (1024, 1024)

api_w, api_h = _pick_inpaint_size(canvas_w, canvas_h)

# 2. Aspect-preserving downsample BOTH flat_img AND mask to API size
flat_resized = _fit_into_canvas(flat_img, (api_w, api_h), bg_rgb=CREAM_BG_RGB)
mask_resized = _fit_into_canvas(mask, (api_w, api_h), bg_rgb=None)  # transparent letterbox

# 3. Call inpaint_with_mask with resized bytes
result_b64 = await provider.inpaint_with_mask(
    image_bytes=flat_resized.tobytes(),
    mask_bytes=mask_resized.tobytes(),
    prompt=inpaint_prompt,
)

# 4. Decode result, fit BACK to original canvas
out_img = Image.open(io.BytesIO(base64.b64decode(result_b64)))
out_img = _fit_into_canvas(out_img, (canvas_w, canvas_h), bg_rgb=CREAM_BG_RGB)

# 5. preserve_alpha re-applies ORIGINAL canvas-size alpha (NOT resized)
rgba = out_img.convert("RGBA")
rgba.putalpha(src_alpha)  # src_alpha is original canvas size

# 6. Save to output_layer_name path
```

Pattern reused from `_fit_into_canvas` (`redraw.py:134-156`). Quality cost: model output is at ~1024-class resolution, upsampled to 4032×3024 — visible as soft edges at pixel-peeping zoom. Acceptable for v0.20 carousel use; v0.21 can add tile-based super-resolution if needed.

## D. Test plan

### D1 — Unit tests (no real provider)

1. **`test_sparse_route_predicate`**: synthetic 1024×1024 RGBA with 3%-area cluster → `sparse=True`. Synthetic with 50%-area solid block → `sparse=False`. Synthetic with 6 disconnected 1.5%-area blobs spread across canvas → `sparse=True` via bbox_fill < 0.5.

2. **`test_mask_is_rgba_no_blur`** + polarity invariant:
```python
mask = _build_inpaint_mask(source_layer)
assert mask.mode == "RGBA"
mask_alpha = np.array(mask.split()[-1])
src_alpha  = np.array(source_layer.split()[-1])
# Polarity invariant — pins the v2 → v2.1 polarity bug from regressing
assert np.all(mask_alpha[src_alpha > 0] == 0),  "subject pixels must be edit zone (mask alpha=0)"
assert np.all(mask_alpha[src_alpha == 0] == 255), "empty pixels must be preserve zone (mask alpha=255)"
# Hard binary — no GaussianBlur
unique = np.unique(mask_alpha)
assert set(unique.tolist()) <= {0, 255}, f"mask must be hard binary; got intermediate values {unique}"
```

3. **`test_capability_fallback_auto`**: mock provider without `inpaint_with_mask`; sparse layer; `route="auto"` → falls back to img2img, no warning. Same with `route="inpaint"` → falls back + warning logged.

4. **`test_dimension_resize_round_trip`**: 4032×3024 RGBA layer; `_redraw_via_inpaint_mask` with stub provider returning a 1024×1024 PNG; output canvas dim must be 4032×3024; alpha must equal original alpha (not resized).

5. **`test_redraw_merged_unchanged`**: existing `redraw_merged` regression — calls `generate` (not `inpaint_with_mask`) regardless of merged alpha sparsity. Pins the B7 deferral.

#### Fixture 1 — single-region sparse

#### Fixture 2 — multi-instance sparse

- File: `docs/visual-specs/2026-04-23-scottish-chinese-fusion/decompose/v0_17_14_native/lanterns.png` (8.05% area, 6 disconnected blobs)
- Pre-fix: documented as "no coherent gongbi geometry" in NOTES.md
- **Acceptance (calibrated 2026-04-27 ship-gate)** — TWO checks:
  - **Structural**: output alpha retains ≥5 spatially distinct connected components (`scipy.ndimage.label`)
  - **Per-component content** (cinnabar-red presence; gold-highlight requirement DROPPED):
    - Mean hue ∈ [0°, 50°] ∪ [330°, 360°] (red / cinnabar / amber — extended

      gpt-image-2 paints "red lantern fill" reliably but "gold accent highlight"

      guarantee. Revisit only if shipping a gold-required style.

  - 18 components detected; 7+ met cinnabar+saturated criteria
  - Structural ≥5 ✓, content ≥4 ✓

## E. Rollout

### v0.20 (this spec)

- `route="auto"` default — auto-routes sparse layers to inpaint, dense layers stay img2img
- `route="img2img"` opt-out (explicit pin to legacy)
- `route="inpaint"` opt-in (force inpaint regardless of sparsity)
- `redraw_merged` UNCHANGED (B7 deferral)
- `background_strategy="cream"` retained as legacy workaround for `route="img2img"`; soft-deprecate via docstring note: "`route='auto'` (v0.20+) makes `cream` unnecessary for sparse layers"
- MCP advisory: log-channel at INFO level when sparse predicate fires:
  ```
  [layers_redraw] sparse_detected=true area_pct=3.3 bbox_fill=0.65 route=inpaint provider=openai
  ```
  Visible to operators via MCP server logs; not in tool return payload (deferred to v0.21 with typed advisory)
- CHANGELOG entry: "Fix: sparse-alpha layers (e.g. small wildflowers, multi-instance lanterns) now redraw correctly via OpenAI mask-aware editing. Default behavior is auto-routing based on a sparsity predicate; pass `route='img2img'` to retain v0.18.x behavior."

### Backward compat

- All v0.18.x callers with no `route` kwarg → get `auto` → dense behavior bit-identical, sparse behavior fixed
- All non-OpenAI providers (gemini, comfyui, mock) → capability fallback → bit-identical to v0.18.x
- All `gpt-image-1` callers → enforced at provider layer; same gate as v0.17.x inpaint
- No env-var kill-switch needed — `route="img2img"` IS the kill-switch

### v0.21 (deferred work)
