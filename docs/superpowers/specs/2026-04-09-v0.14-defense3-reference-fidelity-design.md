# v0.14 "Defense 3" — A-path Reference Image Fidelity

**Date:** 2026-04-09
**Status:** Approved, ready for implementation plan (next session)
**Scope:** Two features bundled: (1) A-path cross-layer style consistency via style_ref, (2) user reference image flows into A-path generation.

## Background

v0.13 shipped the native A-path layered generation (`layered_generate.py`) with keying (Defense 1) and alpha validation (Defense 2). The missing third defense is visual fidelity — ensuring generated layers are stylistically consistent with each other and with user-provided reference images.

The legacy B-path (`layer_generate.py`) already has style reference passing: the first layer's output becomes `reference_image_b64` for subsequent layers. The A-path generates all layers independently in parallel with no visual cross-reference — only text-level context (`sibling_roles` list).

User's framing from a prior session: "keying + validation 之外的'参考图保真'闭环没做".

## Goals

1. **Cross-layer style consistency** — layers generated by the A-path share a coherent visual style, achieved by using the first layer's output as a style anchor for subsequent layers.
2. **User reference image passthrough** — when the user provides a reference image (via `create.py` or pipeline `reference_image_b64`), it influences A-path generation through the same mechanism.

## Non-Goals

- VLM-based semantic content verification (e.g. "does this layer actually contain mountains?") — deferred, high cost, unclear ROI.
- Reference-aware cache keys — intentionally omitted. Adding `reference_image_b64` to cache keys would make cache permanently miss (the reference is the first layer's output, which varies per run). If needed, this is v0.15 scope.
- Changes to the `ImageProvider` Protocol — `reference_image_b64` parameter already exists on all providers.
- Changes to `create.py` — already correctly resolves user reference images.

## Architecture

### Signal chain

```
User reference image (optional, from create.py / PipelineInput)
        ↓
   First layer generation (serial, reference_image_b64 = user reference)
        ↓
   First layer output PNG → base64 = style_ref
        ↓
   Remaining layers generated in parallel (reference_image_b64 = style_ref)
```

When no user reference image is provided, the first layer generates without a reference. Its output still becomes the style anchor for subsequent layers.

### Execution flow change in `layered_generate`

**Before (v0.13):**
```
all layers → asyncio.gather (parallel, no reference)
```

**After (v0.14):**
```
plan[0] → serial generate (with user reference if any)
         ↓ read output PNG → base64 = style_ref
plan[1:] → asyncio.gather (parallel, reference_image_b64 = style_ref)
```

If `plan[0]` fails, fall back to parallel generation without reference (degraded but functional — same as v0.13 behavior).

### 3. `src/vulca/layers/layered_generate.py` — `generate_one_layer`

Add `reference_image_b64: str = ""` to the keyword arguments. Pass it through to `_call_provider_with_retry`.

**Review finding (P1):** `generate_one_layer` must also preserve the raw RGB bytes (before keying) on the returned `LayerOutcome`, so `layered_generate` can use them as style_ref instead of the keyed RGBA. Add an optional `raw_rgb_bytes: bytes | None = None` field to `LayerOutcome` and populate it from `_call_provider_with_retry`'s return value on fresh generation (not on cache hit — cache stores keyed RGBA only).

### 4. `src/vulca/layers/layered_generate.py` — `layered_generate`

Add `reference_image_b64: str = ""` parameter.

Change execution flow:
```python
async def layered_generate(
    *,
    plan: list[LayerInfo],
    ...
    reference_image_b64: str = "",
) -> LayeredResult:
    ...
    sibling_roles = [l.tradition_role or l.name for l in plan]

    # Phase 1: Generate first layer serially (style anchor)
    first = plan[0]
    first_outcome = await generate_one_layer(
        layer=first,
        ...,
        reference_image_b64=reference_image_b64,  # user ref if any
    )

    # Derive style_ref from first layer's RAW output (before keying).
    # Review finding (P1): using the keyed RGBA would pass a partially-
    # transparent image as style reference, which degrades provider output.
    # Store raw RGB bytes during generation and use those instead.
    style_ref = ""
    if first_outcome.ok and first_outcome.raw_rgb_bytes:
        style_ref = base64.b64encode(first_outcome.raw_rgb_bytes).decode()

    # Phase 2: Generate remaining layers in parallel with style_ref
    remaining = plan[1:]
    async def _run(layer: LayerInfo) -> LayerOutcome:
        async with sem:
            try:
                return await generate_one_layer(
                    layer=layer,
                    ...,
                    reference_image_b64=style_ref,
                )
            except ...:
                ...

    rest_outcomes = await asyncio.gather(*(_run(l) for l in remaining))
    outcomes = [first_outcome] + list(rest_outcomes)
    ...
```

If `first_outcome.ok` is False, `style_ref` stays `""` and remaining layers generate without reference — graceful degradation.

**Edge case: single-layer plan.** If `plan` has exactly one layer, `plan[1:]` is empty. No gather runs, no style_ref needed. The single layer generates with the user reference (if any) and returns immediately. This must be covered in tests.

### 5. `src/vulca/pipeline/nodes/layer_generate.py` — `_generate_layers_native`

Pass `reference_image_b64` from pipeline context to `layered_generate`. **Review finding (P0):** `create.py` injects user reference into `node_params["generate"]["reference_image_b64"]`, not top-level context. Must mirror the fallback chain from `generate.py:108-112`:
```python
# Top-level context (multi-round coherence, engine.py:280)
ref_b64 = ctx.get("reference_image_b64") or ctx.get("composite_b64") or ctx.get("image_b64") or ""
# Fallback: node_params (create.py user reference)
if not ref_b64:
    node_params = ctx.get("node_params") or {}
    gen_params = node_params.get("generate") or {}
    ref_b64 = gen_params.get("reference_image_b64", "")
result = await layered_generate(
    ...,
    reference_image_b64=ref_b64,
)
```

### 6. Helper: `_path_to_b64`

Already exists in `layer_generate.py` for B-path. **Review finding (P2):** prefer importing from a shared location rather than duplicating. However, since `layered_generate.py` now uses raw RGB bytes directly (via `LayerOutcome.raw_rgb_bytes`), the `_path_to_b64` helper is no longer needed for the style_ref chain. It may still be useful for other purposes — decide during implementation.

## Cache behavior

`build_cache_key` does NOT include `reference_image_b64`. This is intentional:
- The style_ref is the first layer's output, which varies per run → adding it to cache key would make cache permanently miss.
- When `reference_image_b64` is non-empty, the cache still works: same prompt + same canvas spec → cache hit returns the same keyed PNG regardless of reference. The reference only affects fresh generation.
- This matches B-path's existing behavior (B-path also doesn't cache-key on style_ref).

If reference-aware caching is needed later, it would require hashing the reference image and including the hash in the cache key. Deferred to v0.15.

## Testing strategy

| Test | What it verifies |
|---|---|
| `test_first_layer_serial_then_parallel` | plan[0] generates before plan[1:]; plan[1:] runs in parallel |
| `test_style_ref_passed_to_remaining_layers` | After plan[0] completes, remaining layers receive its output as reference_image_b64 |
| `test_user_reference_passed_to_first_layer` | When user provides reference, plan[0] receives it as reference_image_b64 |
| `test_user_reference_chains_through_first_layer` | User ref → plan[0] → plan[0] output → plan[1:] reference (full chain) |
| `test_first_layer_failure_degrades_gracefully` | plan[0] fails → remaining layers generate without reference (style_ref = "") |
| `test_no_reference_preserves_v013_behavior` | No user ref, no style_ref → all layers generate independently (backwards compat) |
| `test_call_provider_with_retry_passes_reference` | reference_image_b64 flows through retry helper to _call_provider |
| `test_single_layer_plan_no_gather` | Single-layer plan generates directly, no style_ref derivation needed |
| `test_style_ref_uses_raw_rgb_not_keyed_rgba` | style_ref is derived from raw RGB bytes, not the keyed RGBA output |
| Existing tests | Must continue to pass unchanged (default reference_image_b64="" = no-op) |

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| First layer failure → no style_ref → degraded quality | Graceful fallback to no-reference parallel mode. Background layers rarely fail (simple content). |
| Serial first layer adds latency (~3-15s depending on provider load and canvas size) | Acceptable: background is typically the simplest/fastest layer. For an N-layer plan, parallelism drops from N to 1 + (N-1). For 2-layer plans total time roughly doubles vs v0.13. |
| Provider img2img quality varies (some providers produce worse output with reference) | User can opt out by not providing reference. Internal style_ref uses the actual first layer output which is always a valid PNG — not an arbitrary user upload. |
| Cache returns stale result when reference changes | Matches B-path behavior. Cache key doesn't include reference — same prompt hits cache regardless. Acceptable for v0.14. |
| `_path_to_b64` reads potentially large PNG into memory | Layer PNGs are bounded by canvas size (typically 1024x1024 = ~3MB). Acceptable. |

## Out of Scope (recorded for future)

- **VLM semantic verification** — "does this layer contain what the plan described?" Needs extra VLM call per layer, unclear accuracy, high cost. Could be v0.15 or later.
- **Reference-aware cache keys** — hash the reference image, include in cache key. Makes cache more correct but permanently misses for auto-derived style_ref. v0.15.
- **Multi-reference (user ref + style_ref simultaneously)** — would require Protocol change to add a second reference parameter. Not needed: the signal chain (user ref → first layer → style_ref) achieves the same goal through composition.
