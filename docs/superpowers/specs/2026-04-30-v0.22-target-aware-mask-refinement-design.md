# v0.22 Target-Aware Mask Refinement Design

Product/API contract: `docs/superpowers/specs/2026-04-30-redraw-showcase-product-contract-design.md`

## Context

v0.21 fixed the redraw route bug for sparse layers: small alpha regions now route through crop-aware masked edits instead of full-canvas img2img. The roadside showcase exposed the next design defect: `flower_cluster_c` is not sparse enough semantically. Its source alpha covers a broad flower patch, so the edit route preserves color identity but still produces a sticker-like mass.

The root cause is mask granularity, not prompt wording or provider rate limiting. A robust system must detect when an extracted layer is too broad for the user's target intent, split it into semantically useful child masks, redraw those child masks, and verify the result before shipping.

## Goal

Build v0.22 target-aware mask refinement for small visual subjects, starting with bright flower-like targets. The system should identify overly broad masks for tiny/highlight subjects, refine them into child patches using image evidence, route those patches through existing crop-aware masked edits, and expose advisory/quality data that explains what happened.

## Non-Goals

- Do not rebuild the full DINO/SAM orchestrated segmentation pipeline.
- Do not redraw every layer by default.
- Do not require real cloud provider calls in CI.
- Do not make roadside-specific filename hacks. The implementation may use flower-oriented heuristics, but the APIs must be reusable for other small bright targets.

## Desired Behavior

When a layer description or prompt indicates a tiny/flower/highlight target, and the alpha mask has broad geometry, Vulca should:

1. Diagnose the mask as too broad for the target.
2. Refine the mask into multiple child patches using source pixels and alpha geometry.
3. Redraw each child patch independently via the v0.21 crop-aware masked edit path.
4. Paste patches back onto the original canvas using refined alpha masks, not the broad parent mask.
5. Return advisory metadata:
   - `refinement_applied`
   - `refinement_reason`
   - `refined_child_count`
   - `refinement_strategy`
   - `mask_granularity_score`
6. Run local gates that catch over-broad masks, sticker-like large white components, lost target color identity, and background bleed.

## Design

### Target Profile

Introduce a small internal `TargetRefinementProfile` in a new module `vulca.layers.mask_refine`. The first profile is inferred, not user-authored:

- labels/descriptions/instructions containing terms like `flower`, `wildflower`, `blossom`, `petal`, `white`, `tiny`, `small`, `daisy`, `aster`
- target mode: `bright_small_subject`
- expected evidence: high lightness, moderate/low saturation, optional yellow centers, multiple compact components

This is intentionally conservative. If profile inference is uncertain, no refinement occurs.

### Mask Diagnostics

For each candidate layer, compute:

- visible area percentage
- parent bbox width/height/fill
- connected component count
- largest component area ratio
- bright evidence ratio inside alpha
- compact child candidates from luminance-connected components

A layer is too broad when:

- target profile is `bright_small_subject`
- alpha area is large enough to risk a patch edit, or bbox area is visually large
- bright evidence occupies only part of the parent alpha
- candidate child patches exist and cover meaningful target evidence

### Refinement Algorithm

Initial implementation uses deterministic image processing:

1. Crop source RGB and parent alpha to the parent bbox.
2. Convert crop to HSV or HLS.
3. Build a target evidence mask:
   - white-like: high RGB/lightness and low-to-medium saturation
   - optional yellow center evidence: yellow hue with high lightness
   - intersect with parent alpha
4. Clean evidence mask:
   - remove tiny noise
   - close small gaps
   - dilate modestly to include petals
5. Connected components:
   - produce compact boxes
   - merge nearby components into patches
   - cap output count to avoid excessive API calls
6. Return child alpha masks at full canvas size.

### Redraw Integration

`redraw_layer()` should call refinement after loading `src_rgba` and before choosing the geometry route. If refinement returns child masks:

- run `_redraw_via_inpaint_mask()` once per child mask/crop
- use the source/composite crop as context, not an isolated black/transparent layer
- paste each crop back with child alpha
- preserve the parent layer output canvas size
- set output alpha to the union of refined child masks

The existing `SPARSE_PER_INSTANCE` route remains valid. v0.22 refinement happens before geometry route selection and can convert one broad parent layer into several sparse child patches.

### Quality Gates

Extend `redraw_quality.py` with optional refinement-aware metrics:

- `mask_too_broad_for_target`
- `refined_child_count`
- `refined_coverage_pct`
- `white_like_pct`
- `large_white_component`
- `alpha_bbox_expanded`
- `background_bleed`

The gate should not fail dense non-target layers. It should only enforce granularity checks when target profile inference says the layer is a small/bright target.

## Acceptance Criteria

- Unit tests prove `flower_cluster_c`-like broad masks are refined into multiple child masks.
- Unit tests prove non-flower/dense layers are not refined.
- Unit tests prove redraw auto-route uses one provider call per refined child patch.
- Advisory includes refinement fields.
- Existing v0.21 redraw crop tests still pass.
- No real provider credentials are required for CI.

## Showcase Expectation

For the roadside showcase, v0.22 should no longer create a single sticker-like `flower_cluster_c` block. It should redraw smaller flower patches in place while leaving hedge/background pixels mostly untouched. If the source segmentation remains insufficient, the output must say so through advisory metadata rather than silently presenting the result as clean.

Showcase and webapp consumers should not present the transparent redrawn layer as the final image. The user-facing after-image is `source_pasteback_path` when `layers_redraw` can resolve the manifest `source_image`; the editable layer `file` remains the canonical layer asset. The stable display contract is defined in the product/API contract linked at the top of this spec.

The product timeline is:

1. **Precision retouching:** prove exact target control, clean source pasteback, and no surrounding damage.
2. **Local art direction:** use the same target control to make a selected object carry a deliberate visual role.
3. **Scene recomposition:** later, coordinate multiple layer edits under one whole-image concept.

The next public-facing showcase should use Phase 2 framing while depending on Phase 1 reliability. The prompt should describe a user motive, not just replacement pixels:

> The user likes the roadside composition, but this flower cluster feels random and visually flat. Redraw only the selected flower-cluster child masks as a deliberate storybook botanical accent. Preserve the hedge texture, source lighting, and spatial depth. Add rhythmic small white blossoms with subtle warm yellow centers and hand-painted detail. Blend naturally into the hedge; avoid sticker-like edges, oversized flowers, or changes outside the selected masks.
