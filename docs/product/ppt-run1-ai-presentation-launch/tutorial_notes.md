# Tutorial Notes

## Hierarchy

Give each slide one dominant claim, one supporting structure, and one proof object. The top-level claim should be editable text with enough space to breathe. Secondary labels should be shorter than the claim and should explain the diagram, not compete with it.

## Rhythm

Alternate dense explanation slides with lower-density proof or transition slides. Use repeated anchors such as slide numbers, source tags, and proof labels, but vary the main composition so the deck does not feel like ten copies of one template.

## Proof Objects

Every slide needs a visible proof object: a failure-mode list, source-to-rule map, workflow diagram, file inventory, artifact grid, review loop, score matrix, or decision tree. These objects should make the product promise concrete enough for a buyer to inspect.

## Diagram Labels

Diagram labels must be short, horizontal, and editable. Use action verbs for process nodes and nouns for artifacts. Connector arrows should show sequence only when sequence matters; otherwise use grouped panels or lanes to avoid false precision.

## Repair Loops

Treat review as part of the story. The first generation should produce layout JSON and contact sheets; the review prompt should find hierarchy, accessibility, editability, and rendering risks; the repair pass should fix a narrow set of high-impact issues and record what changed.
