# Tutorial Notes

These notes define original design rules for Run 1.5.

## Hierarchy

Each slide should make one claim and one dominant proof object readable from a contact sheet. Secondary panels can explain context, but they must not compete with the main proof object or turn the slide into a document.

## Product Surfaces

Treat the deck as a product lab. Brief fields, memory fields, arm status, render status, provenance status, and review gates should appear as editable product surfaces rather than explanatory paragraphs.

## Editable Proof Objects

Use native PPT shapes, editable text, tables, and simple vector marks for proof objects. Generated or bitmap assets may support atmosphere only when provenance is recorded, and they must not replace editable evidence.

## QA Gates

Every output claim must connect to a reviewable signal: source boundary, design-memory field, comparison score, render check, provenance record, or human approval. Public publishing stays blocked until those signals exist.
