# Vulca Product Strategy Visual Brief

Status: draft-public-blocked

## Visual Mood

Product strategy keynote with an editorial systems feel: one decisive product object per slide, spacious hierarchy, crisp workflow surfaces, and visible handoffs between agent, model, editable artifact, and review. The deck should feel like a product category argument, not a demo log or engineering status report.

## Layout Families

| Slide | Layout Family | Dominant Object | Typography Notes |
| --- | --- | --- | --- |
| 1 | cinematic product thesis | Large horizontal product-loop surface from intent to package. | One short headline, one compact subtitle, no dense labels. |
| 2 | contrast stage | Left/right comparison between prompt-only output and controlled workflow. | Large comparison labels, minimal explanatory copy, strong visual separation. |
| 3 | capability map | System map with agent jobs as lanes and Vulca as the control layer across them. | Use concise lane titles and one-line capability captions. |
| 4 | workflow loop | Circular or stepped loop from usecase to brief, claim spine, visual plan, generation/editing, evaluation, package. | Label each stage with plain product words, not function names. |
| 5 | wedge package | Editable strategy deck package shown as layered business artifact: claims, visuals, gates, handoff. | Give the wedge a large central title and use small proof labels only where needed. |
| 6 | decision board | Roadmap decision board with current base, missing product surface, launch gates, and next bet. | Use clear status language: current, build next, gate before launch. |

## Asset Policy

- Use original diagrams, abstract product surfaces, simple editable shapes, and generated non-branded atmosphere only if all core text remains editable.
- Use local sources for paraphrased product facts and source boundaries, not for copied visuals.
- Source screenshots, logos, copied layouts, long source prose, local evaluation imagery, and raw run artifacts are forbidden.
- Do not use current PPT outputs as hero proof; represent the wedge as a product concept and gated next step.
- Keep provider names and MCP tool names out of main slide copy unless they are simplified into product-language capability groups.
- Treat web research as a future product input with source boundaries, not as an already shipped Vulca surface.

## Anti-Patterns

- Generic SaaS dashboard with repeated cards and tiny labels.
- One-shot image generator interface as the hero product.
- Template-deck tool aesthetic with stock slides and decorative placeholders.
- Dense roadmap table that reads like internal planning notes.
- Visual proof that looks like an engineering report, local test output, or review worksheet.
- Any slide that implies public readiness before render inspection and human review.
