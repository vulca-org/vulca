# Vulca Workflow Demo Visual Brief

Status: draft-public-blocked

## Visual Mood

Polished workflow product demo with editorial pacing: clean product surfaces, restrained but vivid color accents, large proof objects, and visible review states. The deck should feel like a user can follow the workflow from intake to handoff without reading implementation records.

## Layout Families

| Slide | Layout Family | Dominant Object | Typography Notes |
| --- | --- | --- | --- |
| 1 | Editorial cover with live product surface | Usecase intake canvas | Large headline, compact decision metadata, no tiny implementation labels. |
| 2 | Ledger split view | Source boundary ledger | Strong allowed/blocked lane labels, short observation rows, generous spacing. |
| 3 | Board-to-prompt planning view | Claim spine and visual plan board | Six slide slots with one-line claims; prompt handoff as a clear final tile. |
| 4 | Editable canvas cutaway | Native-object slide draft | Labels attach to visible editable objects; no full-slide image illusion. |
| 5 | Review stage board | Evaluation and repair board | Four readable review categories with a single repair decision lane. |
| 6 | Handoff packet composition | Deck package and next decision | Calm close with larger package labels and a decisive next-step state. |

## Asset Policy

- Allowed assets: original native diagrams, abstracted product UI surfaces, generated neutral atmosphere backgrounds, editable SVG accents, and simple icons that do not carry core claims.
- Core slide titles, claims, labels, review gates, and proof annotations must remain editable text.
- Product surfaces should be fictionalized and source-safe: no copied screenshots, logos, source layouts, video frames, raw transcripts, or long source prose.
- Do not reuse prior generated deck copy or prior output visuals as the main proof.
- Source ids may appear in speaker notes or appendix material, not as the public slide story.

## Anti-Patterns

- Implementation-id ribbons or file-path labels dominating the slide.
- Equal-weight card grids across the full deck.
- Tiny status tables that make the workflow feel like a report export.
- A public-ready tone before review and human approval.
- Screenshots used as proof of editability.
- Decorative visuals that do not correspond to a workflow object.
