# Vulca Workflow Demo Generation Prompt

Status: draft-public-blocked

## Prompt

Create a six-slide editable PowerPoint deck for users, design partners, and product reviewers evaluating Vulca as a usecase-to-deck workflow.

The deck decision is: Should the team proceed to generate a workflow demo deck that shows Vulca converting a researched usecase into an editable, reviewable deck package?

Use this thesis: Vulca's demo value is the governed path from researched usecase to editable review packet, not a one-click finished deck.

Use this six-slide arc:
1. A real usecase enters as a business decision, not a loose prompt. Proof object: conceptual usecase intake surface with audience, decision, source packet, and desired output.
2. The first product surface separates allowed observations from claims that stay out. Proof object: conceptual source boundary ledger with allowed and blocked lanes.
3. The workflow turns evidence into a claim spine, visual brief, and generation prompt before slides are made. Proof object: conceptual planning board with six slide slots, claim cards, visual direction, and prompt handoff.
4. The deck draft should be built from editable slide objects, while orchestration stays separate from Vulca MCP capabilities. Proof object: editable slide canvas showing native text, shapes, grouped diagrams, asset callouts, and an agent-side orchestration boundary.
5. The generated deck is judged on editability, provenance, visual quality, and fit to the usecase before any release claim. Proof object: conceptual review board with pass/repair lanes and a visual-quality note.
6. The useful output is a reviewable deck package plus a clear next repair decision. Proof object: conceptual handoff packet with deck file, source ledger, review checklist, and decision state.

Constraints:
- Keep all core text editable.
- Use one dominant proof object per slide.
- Do not copy source visuals, logos, screenshots, layouts, or long prose.
- Do not invent metrics.
- Do not use the shared internal ban-list terms in slide copy.
- Keep public readiness blocked until render and human review pass.
- Do not put source ids, file paths, or implementation labels on main slides.
- Do not claim that Vulca has a dedicated MCP tool for researched deck generation.
- Show current capabilities and agent-side orchestration as separate ideas.
- Label workflow-demo surfaces as conceptual product surfaces, not shipped UI screens.
- Do not render source ids, local paths, or implementation labels on slides; keep claim-source mapping in the companion Markdown files.

## Source Boundary

Allowed use: paraphrase local docs into a product-readable workflow; use source ids for internal review only; represent the work through original product surfaces such as an intake canvas, source ledger, planning board, editable slide canvas, review board, and handoff packet.

Supported today: Vulca MCP and agent-skill capabilities for visual discovery, brainstorming, specification, planning, provider-routed image creation, layer operations, inpainting, image viewing, cultural evaluation, tradition lookup, session archiving, and data sync. The local PPT workflow docs also define a Markdown input pack, editable-native PPT expectations, source boundaries, QA gates, and release-review rules as an agent-run generation process. Keep source mapping in the companion claim spine.

Aspirational or agent-side: one-click researched deck generation as a Vulca MCP tool; automatic public-ready release; finished animation or public video output from motion planning; automatic acceptance of repair recommendations; proof that prior generated deck visuals are polished enough for public distribution. Keep source mapping in the companion claim spine.

Forbidden use: copied source media, source prose, brand marks, source layouts, prior output copy, unsupported performance metrics, hidden source ids as public story structure, or any claim that review gates have already passed.
